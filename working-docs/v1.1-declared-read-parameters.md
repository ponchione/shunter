# v1.1 Declared-Read Parameters

Status: implemented in `v1.1.0-dev`; retained as design/audit notes
Target Shunter release: `v1.1.0`
Baseline checked against: last released `VERSION` = `v1.0.1`
Active development version: `v1.1.0-dev`
Primary surfaces: declared queries, declared live views, protocol declared-read
messages, `@shunter/client`, and TypeScript codegen.

This document started as the validated feature-gap plan around typed dynamic
parameters for declared queries and views. It is intentionally
implementation-facing. Agents working from this note should still inspect live
code first; this file records the direction, constraints, risks, and work
slices, not a replacement for code and tests.

## Version Choice

Use Shunter `v1.1.0` as the target release for this feature set.

Rationale:

- The last released repo version is `v1.0.1`.
- The feature is a normal capability addition after v1 readiness.
- Existing no-parameter declared queries and views can remain source-compatible,
  contract-compatible, and wire-compatible if the implementation uses additive
  metadata and a versioned protocol extension.
- During implementation, the repo version should stay development-stamped as
  `v1.1.0-dev`. Only cut `v1.1.0` when preparing an actual release.

Important qualification: if an implementation changes the meaning or byte
shape of existing stable v1 protocol messages, or changes existing contract
fields in place, retarget that artifact change to a new protocol or contract
version. The Shunter package release can still be `v1.1.0` if it supports both
old and new artifact versions compatibly.

## Original Problem Statement

Before the feature, declared queries and declared live views were named read
surfaces.
They support:

- a declaration name
- executable SQL
- permission metadata
- read-model metadata
- migration metadata
- generated result row decoders when row metadata is available

They do not support app-supplied typed dynamic parameters. A client cannot
cleanly express a stable declared read such as "messages in channel X" without
falling back to raw SQL construction or creating many static declarations.

The immediate TypeScript symptom is that generated declared-read helpers and
the runtime `DeclaredQueryRunner` / `DeclaredViewSubscriber` surfaces do not
have a parameter slot. For example, a generated helper cannot currently expose
an ergonomic call shaped like:

```ts
await queryMessagesByChannel(client.runDeclaredQuery, { channelId });
await subscribeMessagesByChannel(client.subscribeDeclaredView, { channelId }, options);
```

The deeper issue is not TypeScript-only. Shunter lacks a declared-read
parameter contract across the Go module API, contract JSON, SQL compiler,
runtime binding, protocol wire, and SDK/codegen layers.

## Validated Current State

These observations were checked against live code before this document was
written.

### Go Declaration API

`QueryDeclaration` and `ViewDeclaration` are name/SQL/metadata structs. They do
not carry parameter schemas.

Relevant files:

- `module_declarations.go`
- `module.go`

Current declaration shape:

```go
type QueryDeclaration struct {
	Name        string
	SQL         string
	Permissions PermissionMetadata
	ReadModel   ReadModelMetadata
	Migration   MigrationMetadata
}

type ViewDeclaration struct {
	Name        string
	SQL         string
	Permissions PermissionMetadata
	ReadModel   ReadModelMetadata
	Migration   MigrationMetadata
}
```

`Module.Query` and `Module.View` append copied declarations and accept
compatibility-preserving parameter options:

```go
func (m *Module) Query(decl QueryDeclaration, opts ...QueryDeclarationOption) *Module
func (m *Module) View(decl ViewDeclaration, opts ...ViewDeclarationOption) *Module
func WithQueryParameters(product ProductSchema) QueryDeclarationOption
func WithViewParameters(product ProductSchema) ViewDeclarationOption
```

### Runtime Declared Reads

`Runtime.CallQuery` and `Runtime.SubscribeView` accept a declaration name and
declared-read options. The implemented option set now includes ordered app
parameters through `WithDeclaredReadParameters`.

Relevant file: `declared_read.go`

Current shape:

```go
func (r *Runtime) CallQuery(ctx context.Context, name string, opts ...DeclaredReadOption) (DeclaredQueryResult, error)
func (r *Runtime) SubscribeView(ctx context.Context, name string, queryID uint32, opts ...DeclaredReadOption) (DeclaredViewSubscription, error)
```

Current dynamic behavior is limited to caller identity. If a declared read uses
`:sender`, `executableDeclaredRead` recompiles the SQL with the caller identity.
Otherwise it reuses the build-time compiled query.

```go
if entry.UsesCallerIdentity {
	return protocol.CompileSQLQueryString(entry.SQL, r.registry, &caller.Identity, validationOptionsForDeclaredRead(entry.Kind))
}
return entry.compiled.Copy(), nil
```

There is no equivalent for app-supplied parameters.

### SQL Placeholders

The SQL tokenizer recognizes `:name`, but downstream literal parsing only
accepts the exact placeholder `:sender`.

Relevant files:

- `query/sql/parser.go`
- `query/sql/coerce.go`

Current behavior:

- `:sender` becomes `LitSender`.
- Unknown placeholders, different casing, or app-specific names are rejected.
- `:sender` resolves from caller identity and is not an app parameter.

This means a declaration like:

```sql
SELECT * FROM messages WHERE channel_id = :channel_id
```

cannot currently compile.

### Protocol Wire

Declared query and declared view protocol messages carry only correlation IDs
and the declaration name.

Relevant file: `protocol/client_messages.go`

Current message structs:

```go
type DeclaredQueryMsg struct {
	MessageID []byte
	Name      string
}

type SubscribeDeclaredViewMsg struct {
	RequestID uint32
	QueryID   uint32
	Name      string
}
```

Existing client-to-server v1 tags:

- `7`: `DeclaredQuery`
- `8`: `SubscribeDeclaredView`

The existing response messages can probably be reused for parameterized
declared reads. Parameterization changes request admission/execution, not the
shape of one-off query responses or subscription applied/errors.

### Contract JSON

`ModuleContract` currently exports query/view declarations with name, SQL,
row schema, result shape, and passive metadata sections. It has no parameter
schema for declared reads.

Relevant files:

- `contract.go`
- `contract_validate.go`
- `contract_test.go`
- `testdata/v1_module_contract.json`

Current query/view description fields:

```go
type QueryDescription struct {

```

### TypeScript Runtime

The runtime package exposes declared query and view helpers without parameter
support.

Relevant files:

- `typescript/client/src/index.ts`
- `typescript/client/dist/index.d.ts`
- `typescript/client/test/generated-runtime-usage.ts`

Current query runner:

```ts
export interface DeclaredQueryOptions {
  readonly requestId?: RequestID;
  readonly messageId?: Uint8Array;
  readonly signal?: AbortSignal;
}

export type DeclaredQueryRunner<
  Name extends string = string,
  Result = Uint8Array,
> = (name: Name, options?: DeclaredQueryOptions) => Promise<Result>;
```

Current declared view options similarly have request/subscription IDs, abort
signal, return-handle flag, decoders, and callbacks, but no encoded parameter
payload.

### TypeScript Codegen

Generated declared query/view helpers currently call the runtime with only the
declaration name.

Relevant file: `codegen/typescript.go`

Current generated query helper pattern:

```ts
export function queryRecentMessages(runDeclaredQuery: DeclaredQueryRunner): Promise<Uint8Array> {
  return runDeclaredQuery("recent_messages");
}
```

Current generated decoded helper pattern:

```ts
export async function queryRecentMessagesDecoded(
  runDeclaredQuery: DeclaredQueryRunner,
  options: DeclaredQueryDecodeOptions<RecentMessagesQueryRows> = {},
): Promise<DecodedDeclaredQueryResult<typeof queries.recentMessages, RecentMessagesQueryRows>> {
  return queryRecentMessagesResult(await runDeclaredQuery("recent_messages"), options);
}
```

Current generated declared view helper pattern:

```ts
export function subscribeLiveMessageProjection(
  subscribeDeclaredView: DeclaredViewSubscriber,
  options: DeclaredViewSubscriptionOptions<LiveMessageProjectionViewRow> = {},
): Promise<SubscriptionUnsubscribe> {
  const subscribeOptions =
    options.decodeRow === undefined ? { ...options, decodeRow: decodeLiveMessageProjectionViewRow } : options;
  return subscribeDeclaredView("live_message_projection", subscribeOptions);
}
```

## Goals

The completed feature should provide all of the following.

1. App authors can declare typed parameters for declared queries and declared
   live views.

2. Parameter declarations are exported into the module contract in a stable,
   codegen-friendly shape.

3. Shunter validates declared-read SQL against the parameter schema at build
   time where possible.

4. Shunter validates supplied parameter values at call/subscribe time.

5. Protocol clients can execute/subscribe parameterized declared reads without
   constructing raw SQL strings.

6. Generated TypeScript clients expose typed parameter objects and generated
   parameter encoders.

7. No-parameter declared reads continue to work with existing Go call sites,
   existing v1 protocol clients, and existing generated TypeScript helpers.

8. Permission checks, visibility filters, and caller identity behavior continue
   to apply exactly as they do for no-parameter declared reads.

9. Parameter values are treated as user input. They must not be interpolated
   into SQL text strings.

## Non-Goals

These are intentionally out of scope for the first implementation.

- Raw SQL client-side parameter binding for `OneOffQuery`, `SubscribeSingle`,
  or `SubscribeMulti`.
- Dynamic table names, column names, projection names, aggregate function names,
  or `ORDER BY` identifiers.
- Broad SQL compatibility beyond the current Shunter SQL subset.
- Server-side TypeScript or framework cache adapters.
- Stored procedures or read reducers.
- A generic app-level JSON parameter transport. The runtime should use Shunter
  product schemas and BSATN for typed values.
- Changing the behavior of `:sender`. It remains caller identity, not an app
  parameter.

## Proposed User-Facing Model

The feature should be framed as "declared-read parameters", not as generic SQL
string interpolation.

Recommended terms:

- "parameter schema" for declaration metadata
- "params" for generated TypeScript object values
- "encoded params" for BSATN bytes on runtime/protocol surfaces
- "declared-read args" only where code is intentionally mirroring reducer args

### Declaration Example

The examples below use a generic message/channel domain only to show shape.

Preferred Go declaration shape:

```go
mod.Query(shunter.QueryDeclaration{
	Name: "messages_by_channel",
	SQL:  "SELECT * FROM messages WHERE channel_id = :channel_id ORDER BY sent_at DESC LIMIT 50",
	Permissions: shunter.PermissionMetadata{
		Required: []string{"messages:read"},
	},
}, shunter.WithQueryParameters(shunter.ProductSchema{
	Columns: []shunter.ProductColumn{
		{Name: "channel_id", Type: "string"},
	},
}))

mod.View(shunter.ViewDeclaration{
	Name: "live_messages_by_channel",
	SQL:  "SELECT * FROM messages WHERE channel_id = :channel_id",
	Permissions: shunter.PermissionMetadata{
		Required: []string{"messages:subscribe"},
	},
}, shunter.WithViewParameters(shunter.ProductSchema{
	Columns: []shunter.ProductColumn{
		{Name: "channel_id", Type: "string"},
	},
}))
```

Parameter options preserve the exported `QueryDeclaration` and
`ViewDeclaration` literal shape for no-parameter declared reads.

### Local Go Runtime Example

Minimal local runtime call shape:

```go
result, err := rt.CallQuery(
	ctx,
	"messages_by_channel",
	shunter.WithDeclaredReadParameters(types.ProductValue{
		types.NewString(channelID),
	}),
	shunter.WithDeclaredReadPermissions("messages:read"),
)
```

Minimal local view subscription shape:

```go
sub, err := rt.SubscribeView(
	ctx,
	"live_messages_by_channel",
	queryID,
	shunter.WithDeclaredReadParameters(types.ProductValue{
		types.NewString(channelID),
	}),
	shunter.WithDeclaredReadPermissions("messages:subscribe"),
)
```

Recommended first-pass local API:

```go
func WithDeclaredReadParameters(values ProductValue) DeclaredReadOption
```

Optional later convenience:

```go
func WithDeclaredReadNamedParameters(values map[string]Value) DeclaredReadOption
```

The product-value form should be the first implementation because it preserves
declaration order and maps directly to exported product schemas and BSATN. A
map helper can be added later if it proves useful.

## Contract Design

The contract must expose parameter schemas for codegen and compatibility
review.

Recommended JSON field if adding directly to query/view descriptions:

```json
{
  "name": "messages_by_channel",
  "sql": "SELECT * FROM messages WHERE channel_id = :channel_id",
  "parameters": {
    "columns": [
      { "name": "channel_id", "type": "string" }
    ]
  },
  "row_schema": {
    "columns": []
  },
  "result_shape": {
    "kind": "table",
    "table": "messages"
  }
}
```

Recommended Go fields:

```go
type QueryDescription struct {
	Name        string
	SQL         string
	Parameters *ProductSchema
	RowSchema   *ProductSchema
	ResultShape *ReadResultShape
	Permissions PermissionMetadata
	ReadModel   ReadModelMetadata
	Migration   MigrationMetadata
}

type ViewDescription struct {
	Name        string
	SQL         string
	Parameters *ProductSchema
	RowSchema   *ProductSchema
	ResultShape *ReadResultShape
	Permissions PermissionMetadata
	ReadModel   ReadModelMetadata
	Migration   MigrationMetadata
}
```

JSON tags should be `json:"parameters,omitempty"`.

If direct fields are rejected for Go source-compatibility reasons, use a
separate top-level contract section:

```json
{
  "declared_read_parameters": [
    {
      "surface": "query",
      "name": "messages_by_channel",
      "parameters": {
        "columns": [
          { "name": "channel_id", "type": "string" }
        ]
      }
    }
  ]
}
```

Direct fields are easier for codegen and review. A separate section is more
compatible with existing exported Go structs. The implementation owner should
make that tradeoff explicitly before coding.

### Contract Validation Rules

Validation should enforce:

- Parameter schema column names are non-empty and valid under existing product
  schema rules.
- Parameter schema column names are unique.
- `sender` cannot be declared as an app parameter. `:sender` remains reserved.
- Every non-`:sender` placeholder in executable SQL must have a matching
  parameter column.
- Every parameter column in an executable declaration should be used at least
  once, unless the team explicitly decides unused parameters are acceptable for
  forward-compatible client rollout. Default recommendation: reject unused
  parameters to avoid stale contract metadata.
- A parameter used in multiple places must be type-compatible with all target
  columns.
- Metadata-only declarations may carry parameters only if the team wants future
  generated metadata for non-executable reads. Default recommendation: allow
  parameters on metadata-only declarations, but execution still returns
  `ErrDeclaredReadNotExecutable`.
- Parameter schema drift is a contract-diff event. Changing a parameter type,
  nullability, order, or removing a parameter should be breaking. Adding a new
  required parameter should also be breaking for existing generated clients.

### Contract Diff Classification

Suggested policy:

- Add parameter schema to a previously metadata-only or newly added declared
  read: additive if no existing executable helper existed.
- Add required parameter to an existing executable declared read: breaking for
  generated clients because helper signatures change.
- Remove parameter from an executable declared read: breaking for clients using
  the old helper.
- Rename parameter: breaking.
- Reorder product schema columns: breaking for encoded product-value bytes.
- Widen type: classify according to existing product schema compatibility
  rules if available; otherwise breaking.
- Narrow type or nullability: breaking.

## Protocol Design

Do not change the body layout for existing v1 tags `7` and `8`.

Recommended protocol approach for `v1.1.0`:

- Add `ProtocolVersionV2`.
- Add `SubprotocolV2 = "v2.bsatn.shunter"`.
- Keep `ProtocolVersionV1` and `SubprotocolV1 = "v1.bsatn.shunter"` supported.
- Make V2 the current/default protocol for newly generated clients.
- Add parameterized declared-read request tags under V2.
- Reuse existing response tags.

Suggested client-to-server V2 tags:

- `9`: `DeclaredQueryWithParameters`
- `10`: `SubscribeDeclaredViewWithParameters`

Suggested wire bodies:

```text
tag: 9
message_id: bytes
name:       string
params:     bytes     -- BSATN product row encoded according to declaration parameter schema
```

```text
tag: 10
request_id: uint32 LE
query_id:   uint32 LE
name:       string
params:     bytes     -- BSATN product row encoded according to declaration parameter schema
```

Response behavior:

- Query success: existing `OneOffQueryResponse`.
- Query error: existing one-off query error envelope.
- View success: existing `SubscribeSingleApplied`/declared-view applied path.
- View error: existing `SubscriptionError`.
- Transaction deltas: unchanged.

Server behavior:

- A V1 connection must reject V2-only tags as malformed/unsupported for the
  negotiated protocol.
- A V2 connection may still use no-parameter V1-style declared-read tags for
  declarations without parameters.
- A V2 connection sending params to a no-parameter declaration should receive a
  declared-read error, not silently ignore the payload.
- A V2 connection omitting params for a parameterized declaration should
  receive a declared-read error.
- Existing v1 clients continue to work against a `v1.1.0` server when they
  negotiate `v1.bsatn.shunter`.

Alternative approach: add `params?: bytes` to existing tag bodies. This is not
recommended because it changes v1 frame shape and trailing-byte behavior for
stable tags.

Alternative approach: add new tags under the v1 token. This is possible if the
team decides unassigned non-reserved v1 tags may be added, but it weakens the
current "v1 wire is stable" story. A V2 token is cleaner and easier for
compatibility tests.

## SQL Compiler And Binding Design

The SQL layer needs a parameter-aware path. Do not implement this by string
replacement.

### Placeholder Semantics

Reserved:

- `:sender` is caller identity.

App parameters:

- `:name` where `name` exactly matches a declaration parameter column name.
- Matching should be byte-exact and case-sensitive, consistent with Shunter's
  current identifier posture.
- SQL placeholder names should use the contract/product column names, usually
  snake_case.
- TypeScript generated property names should use the existing camel-case
  identifier normalization.

Initial allowed positions:

- literal side of same-relation comparisons, such as `column = :param`
- range predicates, such as `created_at >= :since`
- join/cross-join filter literal positions already supported by current SQL
  predicates
- boolean combinations already supported by current SQL predicates

Initial rejected positions:

- table names
- column names
- aliases
- projection lists
- aggregate names
- `ORDER BY` columns
- `LIMIT` and `OFFSET`
- SQL syntax outside the current parser subset

Later phases may add `LIMIT :limit` and `OFFSET :offset`, but they should not
block the first feature.

### Build-Time Validation

The current declared-read catalog compiles executable SQL at runtime build.
That path assumes either no dynamic values or only caller identity. Parameterized
declared reads need a template validation path.

Required validation outputs:

- referenced tables
- result shape
- row schema
- visibility of `:sender`
- list of app parameter placeholders
- target type expected for each parameter use
- whether any placeholder is unknown
- whether any declared parameter is unused
- whether a parameter is used against incompatible target types

Implementation choices:

1. Add a true parameter-aware compiler template.
   This is the preferred long-term path. The template can validate SQL and
   produce enough metadata for contract export without binding concrete values.

2. Compile with synthetic zero values for parameter placeholders.
   This is faster to implement but fragile. It may hide edge cases, relies on
   fake values being valid for every type, and makes error messages harder to
   reason about.

Recommendation: implement a real parameter-aware template path, even if the
first version internally reuses much of the existing compile code.

### Runtime Binding

At call/subscribe time:

1. Look up the declared read.
2. Check declaration-level permissions.
3. Decode or accept the supplied parameter `ProductValue`.
4. Validate value count, type, and nullability against the parameter schema.
5. Bind parameter values into the compiled SQL template.
6. Resolve `:sender` from caller identity as today.
7. Apply visibility filters as today.
8. Execute the query or register the view.

For no-parameter declarations, keep the existing cached compiled-query path.
For parameterized declarations, either:

- cache the parsed/template form and bind per call, or
- recompile per call from a template AST and concrete parameter values.

Do not cache compiled parameterized predicates globally without including
parameter values in the cache key.

### Subscription Hashing

Parameterized declared views must produce distinct subscription query states
for distinct bound parameter values.

The existing subscription predicate/hash path already includes literal values
in predicates. The implementation should verify:

- bound params become ordinary predicate values before registration
- different param values produce different query hashes
- equal param values and equal caller identity can share query state where
  existing subscription deduping allows it
- `:sender` still uses identity-aware hash behavior
- a query using both `:sender` and app params incorporates both caller identity
  and app parameter values in effective query identity

## Runtime Error Model

Add or reuse errors carefully. Existing declared-read errors include:

- `ErrUnknownDeclaredRead`
- `ErrDeclaredReadNotExecutable`
- `ErrPermissionDenied`
- `ErrInvalidDeclarationSQL`

Recommended new errors:

```go
var ErrDeclaredReadParametersRequired = errors.New("shunter: declared read parameters required")
var ErrDeclaredReadParametersUnexpected = errors.New("shunter: declared read parameters unexpected")
var ErrInvalidDeclaredReadParameters = errors.New("shunter: invalid declared read parameters")
```

Exact names can change, but tests should pin user-visible behavior.

Suggested error cases:

- parameterized declaration called without params
- no-parameter declaration called with params
- encoded parameter bytes fail BSATN decode
- decoded product value has wrong arity
- decoded product value has wrong type
- decoded product value has null where non-null is required
- SQL references undeclared parameter
- declaration schema has unused parameter
- parameter name conflicts with reserved `sender`

Protocol-facing errors should avoid logging or returning full parameter values
unless existing redaction policy explicitly permits them.

## TypeScript Runtime Design

The raw runtime should avoid overloading a plain object as both options and
typed params. Generated helpers can accept typed objects and pass encoded bytes
through options.

Recommended runtime option additions:

```ts
export interface DeclaredQueryOptions {
  readonly requestId?: RequestID;
  readonly messageId?: Uint8Array;
  readonly signal?: AbortSignal;
  readonly params?: Uint8Array;
}

export interface DeclaredViewSubscriptionOptions<Row = unknown> {
  readonly requestId?: RequestID;
  readonly queryId?: QueryID;
  readonly signal?: AbortSignal;
  readonly returnHandle?: boolean;
  readonly decodeRow?: RowDecoder<Row>;
  readonly onInitialRows?: (rows: readonly Row[]) => void;
  readonly onUpdate?: (update: SubscriptionUpdate<Row>) => void;
  readonly onRawUpdate?: RawSubscriptionUpdateCallback;
  readonly params?: Uint8Array;
}
```

Keep `DeclaredQueryRunner` and `DeclaredViewSubscriber` source-compatible by
retaining the same call arity:

```ts
export type DeclaredQueryRunner<
  Name extends string = string,
  Result = Uint8Array,
> = (name: Name, options?: DeclaredQueryOptions) => Promise<Result>;
```

Generated helpers should hide `params` encoding from normal app code.

Runtime validation:

- `params` must be a `Uint8Array` when present.
- If `params` is present and the negotiated subprotocol does not support
  parameterized declared reads, throw a `ShunterProtocolMismatchError` or
  `ShunterValidationError` before sending.
- If `params` is absent, preserve current frame encoding.
- If `params` is present, encode the V2 parameterized frame.

Encoding functions:

```ts
export function encodeDeclaredQueryRequest<Name extends string>(
  name: Name,
  options: DeclaredQueryOptions = {},
): EncodedDeclaredQueryRequest<Name>
```

can either:

- emit old tag/body when `options.params === undefined`
- emit new V2 tag/body when `options.params !== undefined`

or split explicit functions:

```ts
export function encodeDeclaredQueryWithParametersRequest(...)
export function encodeDeclaredViewWithParametersSubscriptionRequest(...)
```

Recommendation: keep explicit lower-level encode functions for golden tests,
and let the high-level client choose based on `params`.

## TypeScript Codegen Design

Codegen should mirror reducer argument support but use "params" naming for
declared reads.

For each executable query/view with a parameter schema:

- generate a `PascalNameParams` interface
- generate a `paramColumns` metadata constant
- generate an `encodePascalNameParams` function
- make raw helper accept `(runner, params, options?)`
- make decoded helper accept `(runner, params, options?)`
- make declared view helper accept `(subscriber, params, options?)`
- make handle helper accept `(subscriber, params, options)`

For no-parameter declarations:

- keep existing helper signatures
- do not require an empty params object

Generated query helper example:

```ts
export interface MessagesByChannelParams {
  channelId: string;
}

const messagesByChannelParamColumns = [
  { name: "channel_id", kind: "string" },
] as const satisfies readonly ShunterBsatnColumn[];

export function encodeMessagesByChannelParams(value: MessagesByChannelParams): Uint8Array {
  return shunterEncodeBsatnProduct([
    value.channelId,
  ], messagesByChannelParamColumns);
}

export function queryMessagesByChannel(
  runDeclaredQuery: DeclaredQueryRunner,
  params: MessagesByChannelParams,
  options: DeclaredQueryOptions = {},
): Promise<Uint8Array> {
  return runDeclaredQuery("messages_by_channel", {
    ...options,
    params: encodeMessagesByChannelParams(params),
  });
}
```

Decoded helper options need careful shape because current decoded helpers use
their second parameter for decode options. For parameterized declarations, use
the second parameter for params and the third for a combined options type.

Recommended generated/runtime type:

```ts
export type DeclaredQueryDecodedRunOptions<
  RowsByName extends object = TableRows,
> = DeclaredQueryOptions & DeclaredQueryDecodeOptions<RowsByName>;
```

Then generated helper:

```ts
export async function queryMessagesByChannelDecoded(
  runDeclaredQuery: DeclaredQueryRunner,
  params: MessagesByChannelParams,
  options: DeclaredQueryDecodedRunOptions<MessagesByChannelQueryRows> = {},
): Promise<DecodedDeclaredQueryResult<typeof queries.messagesByChannel, MessagesByChannelQueryRows>> {
  const queryOptions: DeclaredQueryOptions = {
    requestId: options.requestId,
    messageId: options.messageId,
    signal: options.signal,
    params: encodeMessagesByChannelParams(params),
  };
  return queryMessagesByChannelResult(await runDeclaredQuery("messages_by_channel", queryOptions), options);
}
```

Need to ensure `params` does not leak as a decode option. The decode function
should ignore unknown decode option fields today only if that is already true;
otherwise construct a dedicated decode options object.

## Acceptance Criteria

The feature is complete when all of these are true.

1. A module can declare an executable parameterized query with a typed parameter
   schema.

2. A module can declare an executable parameterized live view with a typed
   parameter schema.

3. `Runtime.ExportContractJSON()` emits parameter metadata deterministically.

4. Contract validation rejects bad parameter metadata and bad SQL/parameter
   combinations.

5. Local `Runtime.CallQuery` can execute a parameterized query with typed
   parameter values.

6. Local `Runtime.SubscribeView` can subscribe a parameterized view with typed
   parameter values.

7. Protocol clients can execute parameterized declared queries over the new
   negotiated protocol.

8. Protocol clients can subscribe parameterized declared views over the new
   negotiated protocol.

9. Existing v1 no-parameter clients still connect and use no-parameter declared
   reads.

10. Generated TypeScript exposes parameter object types and encoders.

11. Generated TypeScript helpers for parameterized reads require params.

12. Generated TypeScript helpers for no-parameter reads do not require params.

13. TypeScript runtime tests, generated usage typechecks, Go targeted tests,
    `go vet` for touched packages, and relevant static analysis pass.

## Open Decisions

These should be resolved before broad implementation starts.

1. Go authoring API:
   Resolved in favor of compatibility-preserving registration options:
   `WithQueryParameters` and `WithViewParameters`.

2. Contract placement:
   Put `parameters` directly on `queries[]` / `views[]`, or add a separate
   top-level parameter metadata section.

3. Protocol evolution:
   This document recommends V2 tags under `v2.bsatn.shunter`. Confirm that
   before implementing wire changes.

4. Placeholder scope:
   First implementation should support predicate literal positions only.
   Confirm whether `LIMIT :limit` / `OFFSET :offset` are needed in v1.1 or can
   wait.

5. Parameter nullability:
   Product schema supports nullable columns elsewhere. Confirm whether nullable
   declared-read parameters are allowed in v1.1 and define SQL comparison
   semantics if they are. Conservative default: reject nullable parameters
   until SQL null semantics are explicitly designed.

6. Metadata-only declarations:
   Decide whether metadata-only declarations may carry parameter schemas.

7. Error redaction:
   Decide exactly how much parameter detail can appear in protocol errors,
   logs, traces, and metrics.

8. Package versions:
   Decide whether `typescript/client/package.json` moves to `1.1.0` with the
   Shunter `v1.1.0` release or remains independently versioned.

## Risks

- Exported Go struct field additions can break external unkeyed composite
  literals.
- Protocol current-version changes will update generated contract metadata and
  golden files.
- Parameterized live views can multiply subscription query states by parameter
  cardinality.
- Recompiling parameterized SQL on every call/subscribe may be acceptable for
  v1.1 but should be benchmarked on common paths.
- Parameter values may include sensitive data. Logging and error messages must
  not leak values casually.
- Ambiguous TypeScript APIs can confuse params with options. Keep raw runtime
  params encoded in options and let generated helpers accept typed objects.
- `:sender` and app params must compose cleanly without changing existing
  caller-identity visibility behavior.

## Suggested Initial Milestone

The smallest useful milestone is:

1. Contract/export can represent parameter schemas.
2. SQL compiler accepts one single-table equality parameter.
3. Local `Runtime.CallQuery` works for that shape.
4. TypeScript codegen emits a typed parameter object and raw query helper.

That milestone does not need protocol V2 or live views yet, but it should be
implemented in a way that does not conflict with the full design above.

Once that path is proven, expand to:

1. protocol V2 parameterized declared queries
2. declared live views
3. joins/ranges/boolean predicates
4. generated decoded helpers and subscription handle helpers
5. full docs and compatibility/support updates
