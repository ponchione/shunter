# OI-003 Recovery and Store Semantics Plan

This is the planning, design, and progress document for OI-003.

`TECH-DEBT.md` is only an index for this campaign. If this file and `TECH-DEBT.md`
disagree, use this file after task-specific user instructions.

## Purpose

OI-003 hardens Shunter's recovery and store semantics so a personally operated
Shunter app can trust restart behavior.

The campaign is about Shunter operational correctness, not byte-format
compatibility with SpacetimeDB. Reference behavior may provide evidence for a
decision, but the final contract should be Shunter-owned and simple enough to
operate.

## Current Scope

Primary goals:

- crash/restart correctness for committed transactions
- deterministic replay from snapshots plus commit log segments
- snapshot compatibility checks with clear failure modes
- compaction safety, including safe retention and deletion boundaries
- clear operator-facing errors for corrupt, incomplete, or incompatible storage
- store/value semantics that keep recovered state equivalent to pre-crash state

Primary code surfaces:

- `types/`
- `bsatn/encode.go`
- `bsatn/decode.go`
- `store/commit.go`
- `store/recovery.go`
- `store/snapshot.go`
- `store/transaction.go`
- `commitlog/changeset_codec.go`
- `commitlog/segment.go`
- `commitlog/replay.go`
- `commitlog/recovery.go`
- `commitlog/snapshot_io.go`
- `commitlog/compaction.go`
- `executor/executor.go`

Adjacent work to keep separate unless a concrete reproducer crosses the
boundary:

- OI-007 scheduler restart and replay-edge behavior
- OI-005 lower-level read-view lifetime discipline
- OI-002 query/subscription correctness
- hosted-runtime V1.5 declaration work

## Design Principles

- Prefer fail-closed behavior when history may be unsafe.
- Accept recovery from clearly bounded incomplete tails when all prior committed
  data is validated.
- Keep all operator-visible failures diagnosable with typed errors or stable
  wrapping context.
- Treat snapshots and logs as a single recovery contract, not independent
  features.
- Do not widen on-disk formats, migration behavior, or public APIs without a
  concrete OI-003 requirement.
- Write the failing or contract-pinning test before changing behavior.

## Locked Decisions

These decisions are the OI-003 campaign contract unless reopened by a concrete
failing example or explicit operator requirement.

### D-001 Durability Acknowledgement Contract

Locked decision:

- V1 `StatusCommitted` is an accepted-commit acknowledgement, not an fsync
  durability acknowledgement.
- A reducer success response may be sent after the reducer has run, the
  transaction has committed to in-memory state, the changeset has been admitted
  to the durability pipeline without silent drop, and the synchronous
  post-commit subscription evaluation path has completed far enough to build
  the response.
- The durable boundary is `DurabilityHandle.WaitUntilDurable(txID)` /
  `DurabilityWorker.DurableTxID()` after the commit-log segment sync completes.
- A crash after `StatusCommitted` but before the durable boundary may lose that
  accepted commit on restart. OI-003 must keep this state explicit and must not
  claim stronger recovery semantics without adding a separate durable-response
  mode.
- Subscription fanout is not part of the storage durability boundary.

### D-002 Recovery Source Of Truth

Locked decision:

- select the newest valid snapshot at or below the durable log horizon
- initialize committed state from the current runtime registry
- restore the selected snapshot when present
- replay contiguous commit-log records newer than that snapshot
- return the highest recovered transaction ID and the append resume plan
- require a complete log from tx 1 when no snapshot is selected
- fail closed on unsafe history gaps, forks, schema mismatch, unsupported
  versions, or sealed-segment corruption

Recovery owns restart state. The commit log and snapshots are not independent
sources of truth; they form one recovery contract.

### D-003 Tail Damage Policy

Locked decision:

- the last segment in ordered segment discovery is the active segment; earlier
  segments are sealed for recovery purposes
- incomplete or corrupt active-segment tail after at least one validated record
  may recover through the validated prefix
- after recovering through a damaged active tail, future writes must use a fresh
  next segment, not append in place
- damage before the first active-segment record fails closed
- corruption in a sealed segment fails closed
- preallocated zero-tail bytes are end-of-stream, not user data

### D-004 Snapshot Compatibility

Locked decision:

- snapshot schema version must match the current registry version
- table IDs, table names, column order, column names, column types,
  nullability, autoincrement flags, index names, index columns, uniqueness, and
  primary-index flags must match the current registry exactly
- v1 nullable snapshot columns are incompatible and should surface a
  `SchemaMismatchError` that wraps `schema.ErrNullableColumn`
- unreadable or corrupt snapshot candidates may be skipped during newest-first
  snapshot selection
- a structurally readable but schema-incompatible snapshot fails closed instead
  of falling back to older snapshots, because it signals that the operator is
  opening storage with the wrong runtime schema
- if no safe compatible base exists, recovery fails closed
- automatic schema migration is out of scope for OI-003

### D-005 Sequence And Autoincrement Recovery

Locked decision:

- recovery must never reuse an already committed autoincrement value
- encoded snapshot sequence metadata and table next-row-ID metadata are restored
  first
- replayed changesets must leave sequence state deterministic from snapshot plus
  log contents
- after snapshot plus replay, each autoincrement sequence must be at least one
  greater than the maximum committed value observed in its autoincrement column
- explicit row values in replay may raise the minimum safe next sequence value
  but must never regress an encoded sequence
- the source of truth is a checked combination: encoded metadata provides the
  intended next value, and observed committed rows provide the lower safety
  bound

### D-006 Compaction Safety

Locked decision:

- only delete sealed segments fully covered by a verified snapshot at or below
  the durable horizon
- retain the boundary segment when it contains records newer than the snapshot
  transaction ID
- always retain the active segment
- delete sidecar offset-index files only with their covered segment
- tolerate missing sidecar index files
- fsync the containing directory after successful deletions where the platform
  supports it
- return the first cleanup failure and stop rather than reporting successful
  compaction after a partial failure

### D-007 Operator Failure Taxonomy

Locked decision:

- keep category sentinels matchable with `errors.Is`: traversal, open,
  durability, snapshot, and index errors
- keep structured details matchable with `errors.As` where fields matter:
  checksum mismatch, history gap, schema mismatch, bad version, unknown record
  type, record/row too large, and offset-index monotonicity
- use stable wrapping context for path, txID, table ID, and segment details at
  the seam where the failure is discovered
- bounded active-tail recovery is represented by the recovery resume plan, not
  by returning an error
- unsupported storage versions, schema incompatibility, unsafe gaps/forks, and
  sealed corruption are fail-closed operator errors

### D-008 Store Value Equivalence

Locked decision:

- row values, table contents, indexes, primary-key uniqueness, and sequence
  state must match the pre-crash committed state at the recovered transaction ID
- table next-row-ID state must also match or be advanced to a safe non-reusing
  value
- recovered rows must not alias mutable decode buffers or changeset inputs
- replay order must be deterministic even when changeset/table encoding order is
  not externally observable
- tests should compare exported state by value, use stable row ordering, verify
  index lookups in addition to table scans, and include sequence / next-ID
  assertions where schemas use autoincrement

### D-009 Storage Format Versioning

Locked decision:

- current Shunter formats remain explicitly versioned
- unknown future versions fail closed with operator-readable errors
- version checks must be pinned across segment headers, changesets, snapshots,
  and schema snapshots
- automatic migration, compatibility shims, and best-effort downgrade behavior
  are out of scope unless a separate migration design is approved

### D-010 Snapshot Completion And Atomicity

Locked decision:

- a snapshot is complete and selectable only when its txID directory has no
  `.lock`, contains a readable `snapshot` file, has a valid header/version/hash,
  and matches the current runtime schema
- a snapshot with `.lock` is incomplete and skipped during selection
- unreadable, truncated, or hash-invalid snapshot candidates may be skipped
  during newest-first selection
- a structurally readable but schema-incompatible snapshot fails closed instead
  of falling back to an older snapshot
- the current direct-write-plus-lock model is the live behavior to inventory,
  but the long-term OI-003 target is the temp-file-plus-rename protocol locked
  in D-017
- snapshot file sync and completion-directory sync failures are operator-visible
  failures; code should not report successful snapshot completion after those
  failures
- cleanup of an incomplete snapshot is best effort; the recovery contract is
  that incomplete snapshots are ignored safely

### D-011 Crash Fault Model

Locked decision:

- OI-003 tests should simulate file artifacts Shunter can realistically observe
  after process crash or interrupted write
- in scope: partial segment record header, payload, and CRC; checksum mismatch;
  zero-filled preallocated tails; corrupt first active record; corrupt active
  tail after a valid prefix; sealed-segment corruption; snapshot lock left
  behind; snapshot hash mismatch; truncated snapshot body; and compaction
  interruption after segment deletion but before sidecar cleanup
- out of scope unless a concrete reproducer requires it: kernel/fsync lies,
  arbitrary torn-sector models, multi-process writers, and disk reordering
  beyond the file artifacts represented in tests
- crash-simulation tests should assert recovery state, resume plan, and
  operator-visible error category/context, not only whether an error occurred

### D-012 Recovery Observability

Locked decision:

- OI-003 should define a structured recovery report shape, but the first
  implementation slices do not need to expose it as public API
- the report shape should be able to represent selected snapshot txID, durable
  log horizon, replayed tx range, recovered highest txID, resume mode, skipped
  snapshot count/reasons, and damaged-tail recovery details
- typed errors and wrapping context remain the primary operator surface until a
  concrete runtime/API slice exposes the report
- no existing public recovery API should be widened solely for observability
  during the initial inventory/test-pin slices

### D-013 Durable Response Mode

Locked decision:

- OI-003 does not add a caller-visible durable-response mode
- V1 `StatusCommitted` remains accepted-but-not-fsync-durable per D-001
- any future mode that waits for `WaitUntilDurable(txID)` before responding is a
  separate runtime/product decision because it changes latency, backpressure,
  and caller semantics

### D-014 Advisory Offset Index

Locked decision:

- per-segment offset indexes are replay accelerators only
- offset indexes are never recovery truth and must not affect recovered state
- missing, corrupt, stale, or unusable offset indexes must fall back to linear
  segment scanning when the segment itself is valid
- offset-index failures should be logged or categorized as index failures where
  they are directly surfaced, but they should not make otherwise recoverable log
  history fail closed
- compaction cleanup follows segment ownership: sidecar indexes are deleted with
  their covered segment, and missing sidecars are tolerated

### D-015 Snapshot TxID Ownership

Locked decision:

- a snapshot txID names the committed state through that txID
- the live snapshot writer validates the committed-state horizon for its txID;
  OI-003 tests also pin that recovery treats the snapshot txID as the replay
  boundary
- a selected snapshot at tx N must not require replaying records at or below N
  to reconstruct state, and recovery must replay only contiguous records newer
  than N
- concurrent commits during snapshot writing must not produce a snapshot whose
  body and txID describe different committed horizons; if live code cannot
  guarantee this, the first implementation slice should either pin the current
  caller discipline or add the narrow synchronization needed to make it true

### D-016 Recovery Report API Placement

Locked decision:

- recovery reporting should become a first-class additive API surface, not only
  logging and not only internal test plumbing
- keep simple recovery APIs simple; add a detailed recovery entrypoint rather
  than forcing every caller to handle a report
- the detailed API should return committed state, recovered txID, resume plan,
  and a `RecoveryReport` or equivalent structured value
- runtime startup should preserve the report for diagnostics and may log a
  concise summary, but logging is a consumer of the report, not the report
  itself
- the report should include selected snapshot txID, durable log horizon,
  replayed tx range, recovered highest txID, resume mode, skipped snapshot
  count/reasons, damaged-tail recovery details, and compaction-relevant segment
  coverage where available
- preserve backward compatibility by making existing simple APIs wrappers around
  the detailed path where practical

### D-017 Snapshot Rename Hardening

Locked decision:

- OI-003 should move snapshot completion to a temp-file-plus-atomic-rename
  protocol rather than leaving direct-write-plus-lock as the long-term target
- the target write sequence is: create snapshot txID directory, create `.lock`,
  write `snapshot.tmp`, fsync `snapshot.tmp`, close it, rename it to `snapshot`,
  fsync the snapshot directory, remove `.lock`, and fsync the snapshot directory
  again
- when creating a new txID directory, fsync the parent snapshot directory where
  the platform permits it
- recovery/listing treats `.lock` and leftover temp files as incomplete snapshot
  evidence; incomplete candidates are ignored safely
- the final selectable artifact remains `snapshot`, so the hardening should not
  require a storage-format migration
- sync/rename/remove failures are operator-visible failures; successful
  completion must not be reported if the snapshot cannot be made durable by the
  target protocol

### D-018 Snapshot Writer TxID Validation

Locked decision:

- caller-owned txID discipline is not acceptable as the permanent contract
- the snapshot writer should eventually receive or verify a committed-state
  horizon so a successful snapshot cannot pair txID N with state from another
  committed horizon
- the target API should make the snapshot horizon explicit, either by writing
  from a snapshot/read-view object that carries its txID or by validating the
  committed state's stable horizon before and after serialization
- concurrent commits during snapshot writing must either be excluded by the
  snapshot source or proven not to change the serialized horizon
- current implementation pins this by recording a committed-state horizon and
  rejecting snapshot writes whose requested txID does not match that horizon
  before file work and under the read lock used for body serialization

### D-019 Compaction Partial-Failure Recovery

Locked decision:

- compaction should be idempotent and resumable in the presence of partial
  cleanup failure
- deleting a covered segment and crashing before deleting its sidecar is safe;
  recovery and future compaction must tolerate orphaned sidecar index files
- compaction must never delete a segment unless a verified snapshot and durable
  horizon make that segment unnecessary for recovery
- partial cleanup failures should return operator-visible errors with path and
  cleanup phase context
- repeated compaction after a partial failure should converge without requiring
  manual cleanup for safe orphaned sidecars or already-deleted covered files
- a separate compaction journal is not required unless inventory finds a
  realistic partial state that cannot be made safe through idempotent deletion
  and orphan tolerance

## Remaining Design Questions

No broad OI-003 design choices are intentionally left open after D-001 through
D-019. The next phase should inventory live code against these locked contracts
and promote concrete gaps into implementation slices.

## Validation Pattern

For code changes, run targeted package tests first, then expand as needed:

```bash
rtk go test ./store ./commitlog ./types ./bsatn ./executor -count=1
rtk go vet ./store ./commitlog ./types ./bsatn ./executor
rtk go test ./... -count=1
```

Run `rtk go fmt` on touched Go packages before finishing.
