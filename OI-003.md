# OI-003 Recovery and Store Semantics Plan

This is the planning, design, and progress document for OI-003.

`TECH-DEBT.md` is only an index for this campaign. If this file and `TECH-DEBT.md`
disagree, use this file after task-specific user instructions.

## Purpose

OI-003 hardens Shunter's recovery and store semantics so a personally operated
Shunter app can trust restart behavior.

The campaign is about Shunter operational correctness, not byte-format
compatibility with SpacetimeDB. Reference behavior may provide evidence for a
decision, but the final contract should be Shunter-owned and simple enough to
operate.

## Current Scope

Primary goals:

- crash/restart correctness for committed transactions
- deterministic replay from snapshots plus commit log segments
- snapshot compatibility checks with clear failure modes
- compaction safety, including safe retention and deletion boundaries
- clear operator-facing errors for corrupt, incomplete, or incompatible storage
- store/value semantics that keep recovered state equivalent to pre-crash state

## Design Principles

- Prefer fail-closed behavior when history may be unsafe.
- Accept recovery from clearly bounded incomplete tails when all prior committed
  data is validated.
- Keep all operator-visible failures diagnosable with typed errors or stable
  wrapping context.
- Treat snapshots and logs as a single recovery contract, not independent
  features.
- Do not widen on-disk formats, migration behavior, or public APIs without a
  concrete OI-003 requirement.
- Write the failing or contract-pinning test before changing behavior.

## Locked Decisions

These decisions are the OI-003 campaign contract unless reopened by a concrete
failing example or explicit operator requirement.

### D-001 Durability Acknowledgement Contract

Locked decision:

- V1 `StatusCommitted` is an accepted-commit acknowledgement, not an fsync
  durability acknowledgement.
- A reducer success response may be sent after the reducer has run, the
  transaction has committed to in-memory state, the changeset has been admitted
  to the durability pipeline without silent drop, and the synchronous
  post-commit subscription evaluation path has completed far enough to build
  the response.
- The durable boundary is `DurabilityHandle.WaitUntilDurable(txID)` /
  `DurabilityWorker.DurableTxID()` after the commit-log segment sync completes.
- A crash after `StatusCommitted` but before the durable boundary may lose that
  accepted commit on restart. OI-003 must keep this state explicit and must not
  claim stronger recovery semantics without adding a separate durable-response
  mode.
- Subscription fanout is not part of the storage durability boundary.

### D-002 Recovery Source Of Truth

Locked decision:

- select the newest valid snapshot at or below the durable log horizon
- initialize committed state from the current runtime registry
- restore the selected snapshot when present
- replay contiguous commit-log records newer than that snapshot
- return the highest recovered transaction ID and the append resume plan
- require a complete log from tx 1 when no snapshot is selected
- fail closed on unsafe history gaps, forks, schema mismatch, unsupported
  versions, or sealed-segment corruption

Recovery owns restart state. The commit log and snapshots are not independent
sources of truth; they form one recovery contract.

### D-003 Tail Damage Policy

Locked decision:

- the last segment in ordered segment discovery is the active segment; earlier
  segments are sealed for recovery purposes
- incomplete or corrupt active-segment tail after at least one validated record
  may recover through the validated prefix
- after recovering through a damaged active tail, future writes must use a fresh
  next segment, not append in place
- damage before the first active-segment record fails closed
- corruption in a sealed segment fails closed
- preallocated zero-tail bytes are end-of-stream, not user data

### D-004 Snapshot Compatibility

Locked decision:

- snapshot schema version must match the current registry version
- table IDs, table names, column order, column names, column types,
  nullability, autoincrement flags, index names, index columns, uniqueness, and
  primary-index flags must match the current registry exactly
- v1 nullable snapshot columns are incompatible and should surface a
  `SchemaMismatchError` that wraps `schema.ErrNullableColumn`
- unreadable or corrupt snapshot candidates may be skipped during newest-first
  snapshot selection
- a structurally readable but schema-incompatible snapshot fails closed instead
  of falling back to older snapshots, because it signals that the operator is
  opening storage with the wrong runtime schema
- if no safe compatible base exists, recovery fails closed
- automatic schema migration is out of scope for OI-003

### D-005 Sequence And Autoincrement Recovery

Locked decision:

- recovery must never reuse an already committed autoincrement value
- encoded snapshot sequence metadata and table next-row-ID metadata are restored
  first
- replayed changesets must leave sequence state deterministic from snapshot plus
  log contents
- after snapshot plus replay, each autoincrement sequence must be at least one
  greater than the maximum committed value observed in its autoincrement column
- explicit row values in replay may raise the minimum safe next sequence value
  but must never regress an encoded sequence
- the source of truth is a checked combination: encoded metadata provides the
  intended next value, and observed committed rows provide the lower safety
  bound

### D-006 Compaction Safety

Locked decision:

- only delete sealed segments fully covered by a verified snapshot at or below
  the durable horizon
- retain the boundary segment when it contains records newer than the snapshot
  transaction ID
- always retain the active segment
- delete sidecar offset-index files only with their covered segment
- tolerate missing sidecar index files
- fsync the containing directory after successful deletions where the platform
  supports it
- return the first cleanup failure and stop rather than reporting successful
  compaction after a partial failure

### D-007 Operator Failure Taxonomy

Locked decision:

- keep category sentinels matchable with `errors.Is`: traversal, open,
  durability, snapshot, and index errors
- keep structured details matchable with `errors.As` where fields matter:
  checksum mismatch, history gap, schema mismatch, bad version, unknown record
  type, record/row too large, and offset-index monotonicity
- use stable wrapping context for path, txID, table ID, and segment details at
  the seam where the failure is discovered
- bounded active-tail recovery is represented by the recovery resume plan, not
  by returning an error
- unsupported storage versions, schema incompatibility, unsafe gaps/forks, and
  sealed corruption are fail-closed operator errors

### D-008 Store Value Equivalence

Locked decision:

- row values, table contents, indexes, primary-key uniqueness, and sequence
  state must match the pre-crash committed state at the recovered transaction ID
- table next-row-ID state must also match or be advanced to a safe non-reusing
  value
- recovered rows must not alias mutable decode buffers or changeset inputs
- replay order must be deterministic even when changeset/table encoding order is
  not externally observable
- tests should compare exported state by value, use stable row ordering, verify
  index lookups in addition to table scans, and include sequence / next-ID
  assertions where schemas use autoincrement

### D-009 Storage Format Versioning

Locked decision:

- current Shunter formats remain explicitly versioned
- unknown future versions fail closed with operator-readable errors
- version checks must be pinned across segment headers, changesets, snapshots,
  and schema snapshots
- automatic migration, compatibility shims, and best-effort downgrade behavior
  are out of scope unless a separate migration design is approved

## Next Design Questions

These remain open after the initial decision lock:

- D-010 snapshot write atomicity: lock the file/lock/hash/rename/fsync sequence
  and incomplete-snapshot cleanup expectations
- D-011 crash fault model: define which write, sync, truncate, and rename
  interruptions OI-003 tests must simulate
- D-012 recovery observability: decide whether recovery should expose a
  structured report beyond `RecoveryResumePlan`
- D-013 durable-response mode: decide whether Shunter should later offer a
  caller-visible mode that waits for `WaitUntilDurable(txID)` before responding
