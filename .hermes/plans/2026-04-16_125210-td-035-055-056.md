# TD-035 / TD-055 / TD-056 Implementation Plan

> For Hermes: use test-driven-development for each behavior change before touching production code.

Goal: Land the next narrow executor/store hardening slice covering scheduler cancel failure semantics, lifecycle delete-during-iteration safety, and snapshot leak / commit-block hazard.

Architecture: Keep the slice narrowly scoped to existing executor/store contracts unless a public-surface change is clearly justified by the bug. Prefer targeted safety improvements plus regression tests over broader refactors. Land all three fixes together because they touch adjacent operational safety edges in executor/store behavior.

Tech stack: Go, package-local unit/regression tests, rtk-prefixed build/test commands.

---

## Current context / assumptions

- TECH-DEBT.md names the targets explicitly:
  - TD-035: `store/snapshot.go` leaked snapshot can block commits forever.
  - TD-055: `executor/scheduler.go` cancel failure is indistinguishable from missing row.
  - TD-056: `executor/lifecycle.go` deletes during `ScanTable` iteration.
- Relevant decomposition/spec surfaces:
  - `docs/decomposition/003-executor/epic-6-scheduled-reducers/story-6.2-transactional-schedule.md`
  - `docs/decomposition/003-executor/EPICS.md` Epic 7 lifecycle flow
  - `docs/decomposition/001-store/epic-7-read-only-snapshots/story-7.2-snapshot-concurrency.md`
- Existing tests already cover basic happy paths for scheduler, lifecycle, and snapshots; this slice should extend those tests to prove the hazards are addressed.

## Proposed approach

1. TD-055: promote `Cancel` to return `(bool, error)` so callers can distinguish “missing row” from an operational failure. Update the reducer-facing scheduler interface and all local tests/stubs accordingly.
2. TD-056: make `deleteSysClientsRow` two-phase: scan to capture the target RowID, then perform delete after iteration completes. Add a focused regression test that exercises the helper against committed state.
3. TD-035: harden `CommittedSnapshot` with louder contract enforcement and leak mitigation:
   - capture creation stack / metadata for diagnostics
   - install a finalizer that auto-closes leaked snapshots and logs the leak site
   - clear the finalizer on explicit `Close`
   - add regression coverage that leaked snapshots eventually stop blocking commit after GC, without weakening the documented requirement to close promptly

## Risks / tradeoffs

- Changing `ReducerScheduler.Cancel` is a public contract change; if that feels too broad once edits begin, fall back to the narrower “log the delete error before returning false” approach and document why.
- Finalizer-based leak mitigation is best-effort, not prompt resource release; comments/tests must still emphasize explicit `Close` as mandatory for bounded commit latency.
- GC/finalizer tests can be flaky; keep them narrowly designed and tolerant of repeated `runtime.GC()` polling rather than timing-sensitive sleeps.

## Done criteria

- Cancel semantics distinguish “missing row” from real delete failure.
- `deleteSysClientsRow` no longer mutates during iteration.
- Leaked snapshots are harder to miss and no longer create an indefinite commit wedge once unreachable.
- Targeted tests plus repo-wide build/test pass.
- Diff remains narrow to TD-035/055/056 only.
